# v1.1 — Fallbacks & undo — Design Spec

**Status:** Approved — ready for plan
**Tracking issue:** TBD — open three GitHub issues (one per core item) at plan time, reference back to umbrella issue #36 for the deferred-from-v1.0 lineage.

## Goal

Ship v1.1.0 of Pantry Tracker as the smallest coherent release after v1.0 (2026-05-18). Three core items, all either user-evidenced or one-commit hygiene; no schema migration risk; no R8 minify risk. One optional stretch (accessibility audit).

**The three items:**

1. **Non-food OFF-miss fallback** — query Open Beauty Facts, Open Pet Food Facts, Open Products Facts in sequence when world.openfoodfacts.org returns 404. Closes the gap documented in commit `95c7b9f` ("non-food OFF-miss as v1.1+ out-of-scope item").
2. **SR-24 — Ktor 256 KB body cap** — pure hygiene, last unchecked item under "Hygiene / documentation" in issue #36.
3. **In-app delete-undo snackbar** — replace the "Cannot be undone in v1" copy on `DeleteConfirmDialog`. Closes TD-7 in arc42 §11.

**Stretch (contingent):** WCAG 2.1 AA contrast audit of the fern theme. Pulled in only if items 1–3 merge with > 3 days slack before the tag.

## Non-goals

- **No R8 minify (SR-9).** R8 deserves a dedicated milestone where the only acceptance criterion is "minified APK passes full manual UAT". Bundling it with feature work means any regression is ambiguous in origin (feature vs minifier).
- **No Room schema export (SR-17) and no expiry-date field on `Product`.** SR-17 + first schema bump are a natural pair — both wait for a v1.2 milestone whose entire purpose is "first DB migration done right".
- **No localization, no `strings.xml` extraction.** Deferred per arc42 TD-2 — no German-speaker pressure yet.
- **No crash reporter.** Privacy charter holds; any crash reporter would need to be self-hosted to comply, which is way over a v1.1 budget.
- **No batch-scan, no Wear OS, no animation polish, no tablet/foldable adaptive layouts.** All deferred per CHANGELOG "Out of scope for v1.0 — planned for v1.1+".
- **No manifest changes, no permission changes, no `targetSdk`/`minSdk` bump.** v1.1 is purely application-layer.

## Architecture

Three independent slices, each isolatable to its own PR.

### Item 1 — Non-food OFF-miss fallback

**Current state.** `data/remote/OffApiClient.lookup(barcode)` queries `https://world.openfoodfacts.org/api/v2/product/<barcode>.json`. Every failure path (HTTP non-2xx, IOException, parse error) collapses to a `null` return; the caller (`ScanViewModel`) treats `null` as "drop to manual entry sheet". This invariant is documented in the file's KDoc and load-bearing — the spec keeps it.

**New behaviour.** Walk a fixed-order host chain on `404`:

```
world.openfoodfacts.org      → food (status quo)
world.openbeautyfacts.org    → cosmetics
world.openpetfoodfacts.org   → pet food
world.openproductsfacts.org  → everything else
```

All four are OpenFacts sister projects with the same v2 API surface and the same `OffApiEnvelope` / `OffProduct` JSON schema. Same `OFF_FIELDS` query param. Same `User-Agent`. No parser changes expected; this assumption is validated at implementation time with a real sample request from each host (see implementation plan).

**Internal result type.** Introduce a `private sealed interface` inside `OffApiClient`:

```kotlin
private sealed interface HostResult {
    data class Found(val product: OffProduct) : HostResult
    data object NotFound : HostResult                // walk to next host
    data object Error : HostResult                   // stop walking; outer null
}
```

**Per-host mapping** (inside the existing `try { … } catch (…)` block):

| Source | `HostResult` |
|---|---|
| `response.status == 404` | `NotFound` |
| `response.status.isSuccess()` and `envelope.status == 1` | `Found(envelope.product)` |
| `response.status.isSuccess()` and `envelope.status != 1` | `NotFound` (OFF's "product not in DB" sentinel) |
| Any other non-success status (4xx other than 404, 5xx) | `Error` |
| `IOException` (network, timeout, DNS) | `Error` |
| `JsonConvertException`, `SerializationException`, `IllegalArgumentException` | `Error` |
| `CancellationException` | rethrown (unchanged) |

**Chain logic** (replaces the body of the public `lookup(barcode)`; introduces a new `private suspend fun lookupOnce(host: String, barcode: String): HostResult` helper that contains the current single-host request body — URL building, `httpClient.get(url)`, the existing `try/catch` cascade — but returns `HostResult` instead of `OffProduct?`):

```kotlin
for (host in OFF_HOSTS) {
    when (val r = lookupOnce(host, barcode)) {
        is HostResult.Found -> return r.product
        HostResult.NotFound -> continue
        HostResult.Error    -> return null   // fail fast; do not walk past a real fault
    }
}
return null                                  // all four NotFound
```

**Why fail fast on `Error` instead of walking past it.** A 500 from OFF means *that server* is sick — it does NOT imply "the barcode might exist on Beauty Facts instead". Walking on errors multiplies one server's downtime by 4 (worst case 4 × 8 s = 32 s of timeouts), produces extra DNS+TLS load for no information gain, and degrades the user-visible "Couldn't reach OFF" error UX into "app hangs for 30 seconds, then shows manual sheet". The 404-only contract caps the chain's worst-case observable latency at "one slow OFF request, then three fast 404s".

**Public API contract.** `OffApiClient.lookup(barcode): OffProduct?` is **unchanged** — same nullable return, same `null` on all failure modes. `ScanViewModel`, `ProductRepository`, and existing tests need zero adjustment.

**Constants change** (`OffApiClient` companion):

```kotlin
private val OFF_HOSTS = listOf(
    "https://world.openfoodfacts.org/",
    "https://world.openbeautyfacts.org/",
    "https://world.openpetfoodfacts.org/",
    "https://world.openproductsfacts.org/",
)
```

The existing `OFF_BASE_URL` constant is **replaced** by `OFF_HOSTS` and removed. `OFF_FIELDS`, `BARCODE_PATTERN`, `TIMEOUT_MILLIS`, `USER_AGENT`, `defaultClient()` are all unchanged.

**Privacy doc updates** (required, single edit each):

- `CHANGELOG.md` — Privacy section currently says "single OFF lookup". List all four hostnames.
- `README.md` — same wording, same edit.
- `docs/architecture/08-crosscutting-concepts.md` §8.9 — list all four; note that walking on 404-only caps the per-scan host count at 1 in the happy path.

**Tests** (`OffApiClientTest`, JUnit + Ktor `MockEngine`, no Robolectric needed):

| Case | Setup | Expectation |
|---|---|---|
| OFF hit | OFF→200 | Returns product. Beauty/PetFood/Products **never called** (assert via `MockEngine` history). |
| OFF miss, Beauty hit | OFF→404, Beauty→200 | Returns Beauty product. PetFood/Products never called. |
| Two misses, PetFood hit | OFF→404, Beauty→404, PetFood→200 | Returns PetFood product. Products never called. |
| All four miss | OFF→404, Beauty→404, PetFood→404, Products→404 | Returns `null`. |
| OFF 5xx, fail fast | OFF→500 | Returns `null`. Beauty/PetFood/Products **never called** (regression guard for the "don't walk on error" rule). |
| OFF 404, Beauty timeout | OFF→404, Beauty→IOException timeout | Returns `null`. PetFood/Products never called. |
| OFF status==1 with `envelope.status != 1` | OFF→200 + `status: 0` JSON | Treated as `NotFound`, walks to Beauty. |
| Cancellation | OFF→200 but coroutine cancelled mid-flight | `CancellationException` propagates. |

### Item 2 — SR-24 Ktor 256 KB body cap

**Why.** Defence-in-depth against a hostile or accidentally-huge response. OFF single-product JSON observed at 5–20 KB; 256 KB is a 10× safety factor. Today, a 1 MB response would parse all the way through Ktor's `body<OffApiEnvelope>()` before timing out at the parse stage — slow and memory-hungry. With a body cap, the failure is bounded at 256 KB.

**Pattern.** Add `install(HttpResponseValidator) { … }` to the `defaultClient()` block:

```kotlin
install(HttpResponseValidator) {
    validateResponse { response ->
        val contentLength = response.headers[HttpHeaders.ContentLength]?.toLongOrNull()
        if (contentLength != null && contentLength > MAX_BODY_BYTES) {
            throw OversizedResponseException(contentLength)
        }
        // Note: bodies without Content-Length (chunked) are not size-checked at
        // the header stage. The Ktor v2 `body<T>()` flow uses ContentNegotiation,
        // which reads the entire payload into memory before parse — so a
        // server-side chunked response larger than MAX_BODY_BYTES would still
        // OOM. Acceptable for v1.1: OFF's CDN sends Content-Length for product
        // responses, so the happy/hostile path we actually care about is
        // covered. Streaming-counter enforcement is a v1.2+ consideration only
        // if we ever switch to a streaming parser.
    }
}
```

**Constants & types** (companion + same file):

```kotlin
private const val MAX_BODY_BYTES = 256L * 1024L

private class OversizedResponseException(size: Long) :
    IOException("OFF response body exceeds cap: $size bytes")
```

Extending `IOException` is deliberate — the existing `catch (e: IOException)` arm in `lookup()` (line 95 today) already maps it to `null` (`HostResult.Error` after item 1 lands). No new catch needed. Logged at WARN with the size for forensics.

**Test** (extends `OffApiClientTest`, `MockEngine`):

- Synthetic response with `Content-Length: 1048576` (1 MB) → `lookup()` returns `null` within < 500 ms (assertion: latch fires *before* parse would have).
- Synthetic response with `Content-Length: 262144` (exactly 256 KB) → allowed; parser sees it; product returned if envelope valid.
- Synthetic response with `Content-Length: 262145` → rejected.

### Item 3 — Delete-undo snackbar

**Current.** `DeleteConfirmDialog` reads "Cannot be undone in v1." After confirm, `repo.delete(product)` removes the row and the UI snaps back to the home list with no further affordance.

**New flow.**

1. User confirms delete in the existing dialog.
2. `HomeViewModel.delete(product)` calls `repo.delete(product)` (synchronous from the VM's perspective — repo handles the suspend).
3. VM emits a one-shot `SnackbarEvent.Deleted(product)` through a `Channel<SnackbarEvent>` exposed as `Flow`. The captured `Product` instance carries every field needed to restore — including `id: Long` (preserved post-delete because `@PrimaryKey(autoGenerate = true)` accepts non-zero ids).
4. `HomeScreen` collects the channel inside `LaunchedEffect`, calls `snackbarHostState.showSnackbar("Item deleted", actionLabel = "UNDO", duration = SnackbarDuration.Short)`.
5. If the result is `SnackbarResult.ActionPerformed`, screen calls `viewModel.undoDelete(product)` which calls `repo.insert(product)` with the captured instance. Same `id` restored; barcode unique-index preserved.
6. If the result is `SnackbarResult.Dismissed`, nothing further happens — the row is already gone from the DB.

**Copy change** in `DeleteConfirmDialog`:

| Before | After |
|---|---|
| "This will permanently remove *<name>* from your pantry. Cannot be undone in v1." | "Remove *<name>* from your pantry?" |

The body shrinks because the new undo affordance speaks for itself — the dialog no longer needs to apologise for the irreversibility.

**Why "delete immediately, restore on UNDO" over alternatives.**

- **Soft-delete (`isDeleted` column).** Requires schema change → couples item 3 to SR-17 → breaks milestone separation. Rejected.
- **Buffered delete (hold row in memory until snackbar dismisses, then commit).** Fragile across recomposition, back navigation, process death. Rejected.
- **Immediate delete + closure-captured restore.** Works because (a) `Product.id` survives the round-trip, (b) the unique-index on `barcode` makes any race ("user re-scans same product while snackbar visible") fail loudly through the existing repo error path, and (c) the captured `Product` is GC'd once the snackbar closure goes out of scope, so there's no "stale last-deleted" state to manage. Adopted.

**Edge case — second delete during snackbar window.** User deletes A, snackbar shows, user deletes B before tapping UNDO. Material 3 behaviour: showing a new snackbar dismisses the previous one. Result: A's UNDO is lost (closure unreachable), A stays deleted; B's snackbar takes over with B's UNDO. Acceptable — matches Gmail / Drive web UX for multi-delete-with-undo.

**Edge case — app killed during snackbar window.** Row is already gone from DB at process death. Restore is impossible. Acceptable; the snackbar is a 4-second window and process death there is a force-quit, not a normal lifecycle event.

**Tests:**

- `HomeViewModelTest` (JUnit + Turbine):
  - `delete(product)` → repo `delete` called; `snackbarEvents` flow emits `SnackbarEvent.Deleted(product)` with the same instance.
  - `undoDelete(product)` → repo `insert(product)` called with all fields preserved including `id`.
  - Sequential deletes (A then B before any UNDO) → both repo `delete` calls fired, two separate `SnackbarEvent.Deleted` emissions (channel doesn't drop).
- `HappyPathUatTest` Compose `androidTest` extension:
  - Add an item, delete it, tap UNDO in the snackbar, assert item back in list with `onNodeWithText(name).assertIsDisplayed()` (Compose member, no bare `assert()` — see CLAUDE.md "Compose `androidTest` asserts").
  - Add an item, delete it, wait for snackbar to dismiss (`composeTestRule.mainClock.advanceTimeBy(5_000)`), assert item NOT in list.

### Item 4 (stretch) — Fern theme WCAG audit

**Trigger.** Only pulled into the milestone if items 1–3 are merged with > 3 days slack before the planned v1.1.0 tag date. Otherwise it punts to v1.2 untouched.

**Mechanics.** For every text/background colour pair in `app/src/main/java/de/docgerdsoft/pantrytracker/ui/theme/Color.kt` (light + dark schemes), compute the WCAG 2.1 contrast ratio using `androidx.compose.ui.graphics.Color.luminance()`. Required: 4.5:1 for normal text, 3:1 for large text (≥ 18pt or 14pt bold).

**Pairs to audit** (non-exhaustive — sweep finds the rest by walking `MaterialTheme.colorScheme` usages):

- `onPrimary` on `primary`
- `onSurface` on `surface`
- `onSurfaceVariant` on `surface`
- `onError` on `error`
- `onPrimaryContainer` on `primaryContainer`

**Outcomes:**

- **All pass:** add a row to `docs/architecture/10-quality-requirements.md` documenting "WCAG 2.1 AA verified as of 2026-05-??" with the computed ratios. Zero code change.
- **One or more fail:** adjust the offending token in `Color.kt` and re-run. If the fix is non-obvious (e.g., the fern primary needs to shift hue to keep readable on dark surfaces), defer the actual token change to v1.2 and document the finding as a known issue in `docs/architecture/11-risks-and-technical-debt.md`.

- `feat/v1.1-off-fallback-chain` — item 1 (largest, lands first)

Each PR references `Refs #<new-issue>` per the project's "every PR needs an issue link" rule (CLAUDE.md memory: `feedback_pr_needs_issue_link`). Each goes through the multi-agent review cycle per `feedback_always_pr_review`, with inline-thread findings posted and resolved via the `post-finding` skill.

**Release-time PR.** A final `release/v1.1-prep` PR bumps `versionName = "1.1.0"` and `versionCode` in `app/build.gradle.kts`, drafts the `CHANGELOG.md` v1.1.0 section, and stamps the date. Same keystore as v1.0 (per memory `project_m6_resume_point` — cert SHA-256 `ec9a4bb8…b3d9` is lifetime identity; v1.1 stays on it because it's a normal minor version, not a v1.0.x patch).

**Per-release dependency lockfile commit.** Per SHIPPING.md "Dependency lockfile at release tags", regenerate `app/gradle.lockfile` and force-commit it in a dedicated `chore(release): lock dependencies for v1.1.0` commit immediately before the tag. This is the same procedure as v1.0.

**Detekt / lint.** No new suppressions expected. The `OffApiClient.lookup()` cyclomatic complexity is already at the rule's limit; if introducing `lookupOnce(host, barcode)` as a private helper pushes it over, decompose further — do not add `@Suppress`.

**CI.** No workflow changes. The existing CI runs `assembleDebug`, `testDebugUnitTest`, `lintDebug`, `assembleDebugAndroidTest`, Detekt, Gitleaks, OSV-Scanner — all four are sufficient for v1.1.

- **Item 1.** `OffApiClientTest` covers all eight cases in the table above. The public `lookup(barcode): OffProduct?` contract is unchanged (existing `ScanViewModelTest`, `ProductRepositoryImplTest` need zero edits and continue to pass). Privacy copy updated in `CHANGELOG.md`, `README.md`, arc42 §8.9. A real-device scan of a non-food barcode (e.g., a shampoo bottle) resolves through Open Beauty Facts and the product appears in the pantry list — manual UAT entry.
- **Item 2.** Body-cap test (1 MB synthetic) fires < 500 ms. Boundary tests at exactly 256 KB and 256 KB + 1 byte behave as specified.
- **Item 3.** ViewModel tests cover the happy path, UNDO restore, snackbar dismissal, and back-to-back delete. `HappyPathUatTest` extension passes on the project's CI Compose harness. Delete-confirm dialog copy updated; old "Cannot be undone in v1" string removed (grep `-r "Cannot be undone"` returns zero hits).
- **Item 4 (if pulled in).** Either all contrast ratios pass and arc42 §10 documents it, or a known-issue entry lands in arc42 §11 deferring a colour-token fix to v1.2.

Signed APK on GitHub Releases. Manual UAT pass on a real device per the `docs/uat/v1-uat-checklist.md` smoke list (re-run the existing v1.0 checklist, extended for items 1 + 3).

## Why now

v1.0 shipped 2026-05-18. The single piece of real-world friction we already have evidence of (commit `95c7b9f`) is the non-food OFF-miss — and the user is going to hit it within days of using the app on their own pantry. SR-24 is hygiene that pairs naturally because it touches the same `OffApiClient` file. The delete-undo snackbar removes a paper-cut documented as TD-7 in arc42. All three are independent, all three are bounded, none of the three touch the schema or R8 — so v1.1 ships fast and keeps the two high-risk milestones (R8, first DB migration) cleanly siloed for v1.2 and v1.3.

## Test strategy

- **JVM unit tests (existing infrastructure).** Items 1 and 2 entirely covered here. JUnit 4, Ktor `MockEngine`, Turbine — same stack as M0–M6.
- **Compose `androidTest`.** Item 3's UNDO restore exercised via the existing `HappyPathUatTest` happy-path harness, extended by ~30 lines.
- **Manual UAT.** Real-device scan of a non-food barcode for item 1 (no automated test can verify the actual sister-host JSON schema matches OFF's). Visual smoke for the dialog copy change in item 3. WCAG ratio computation is mechanical for item 4 if pulled in.
- **No new test infrastructure.** Everything reuses the M0–M6 toolchain.
